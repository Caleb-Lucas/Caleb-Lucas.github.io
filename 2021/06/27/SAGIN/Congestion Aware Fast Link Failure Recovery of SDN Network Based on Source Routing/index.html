基于源路由的SDN网络的阻塞感知链路故障恢复方法

发表于 2021-06-27 更新于 2021-06-28 分类于 SAGIN

概述

  • 软件定义网络(Software Defined Network, SDN)中控制平面和数据平面的分离使得其在灵活控制网络行为的同时，也由于故障点到控制器之间的时延给链路故障恢复带来了一些不便。为了避免延迟和丢包，在出现故障时，使用预先定义的备份路径对中断的流进行重路由。但是，这些备份路径的构建和维护开销较大，难以根据网络状态动态构建备份路径，避免重路由过程中的拥塞。为了实现基于拥塞感知的链路故障快速恢复，提出了一种通过源路由和逐跳标记为每条链路安装多条备份路径，并在故障点将流分散到不同路径以避免拥塞的方法。通过实验和仿真对该方法的性能进行了评估，结果表明该方法可以在非常低的开销下实现SDN网络中拥塞感知的快速链路故障恢复。
  • 为了满足载波级网络恢复时间的要求，提出了SDN[1]~[5]中实现近似零损耗和链路故障快速恢复的几种主动机制。在主动机制中，使用端到端或每跳预先计划的备份路径，保证当链路发生故障时，故障点附近的交换机可以立即将流重新路由通过备份路径到达目的地，而无需控制器的决定。实验结果表明，与传统方法相比，该方法可以获得更短的恢复时间。然而，对于主动机制仍然存在一些问题:首先，它可能需要大量流条目来构建备份路径，并需要复杂的机制来保持它们的活动;其次，预先规划的备份路径一旦构建好，不便于修改、删除和重建，难以根据网络状态自适应调整备份路径，因此在对中断的流进行重路由时可能会造成拥塞。
  • 综上所述，故障恢复方法要实现三个目标:(1)快速:对于载波级网络的关键任务要求，必须尽可能快地检测和定位故障点，并对链路故障作出反应。故障恢复总时间应小于50ms;(2)低开销:备份路径消耗的流项尽可能少，使网络能够接受更多的流;(3)拥塞感知:备份路径规划和流重路由过程应考虑网络状态，避免网络其他部分拥塞。
  • 关于目标(1)和(2)[1]~[7]的研究已经有很多，但很少有研究关注到重路由过程中的拥塞问题。在传统的方式下，为了避免[8]拥塞导致的丢包和业务下降，需要在备份路径上为被保护的流预留带宽。但在实际应用中，这种方法可能会造成网络资源的浪费，且部署复杂。为了实现上述三个目标，本文提出了一种感知拥塞的SDN链路故障快速恢复方法。**该方法利用源路由控制网络边缘流的路由。*控制器根据预定义的每条链路的多备份路径和每条流的带宽需求，在报文的源路由报头中插入每一跳备份路径标签，以识别每一跳流的具体备份路径。*通过这种机制，中断流在重路由过程中通过多个备份路径传输，从而避免拥塞。当发生故障时，故障点附近的交换机根据备份路径标签更新报文的源路由报头，报文就可以绕过故障点。该方法利用了源路由的拓扑独立特性，实现了备份路径规划和备份路径选择的解耦。因此，它可以根据网络状态实现备份路径间的流动态分配。
  • 与传统方法相比，我们的方法主要有四个优点:一是可以在不涉及控制器的情况下进行局部链路故障恢复，从而使恢复时间最小化;其次，它可以实现相当低的流量表项消耗，因为它将路由信息存储在包头而不是安装在交换机的流量表项中。第三，它考虑了大多数传统方法忽略的重路由过程中的拥塞问题，可以显著降低拥塞的可能性。第四，既可以处理单链路故障，也可以处理单节点故障。本文的主要贡献如下:(1)提出了一种新的基于源路由的SDN网络体系结构和实现;(2)提出了一种基于源路由SDN的拥塞感知链路故障快速恢复方法;(3)给出了备份路径规划和选择问题的形式化和求解方法;(4)分析了该方法的开销，并对其性能进行了评价。

相关研究

链路故障检测
  • 链路故障恢复时间由故障检测时间和响应时间两部分组成。为了实现链路故障的快速恢复，在故障发生后立即检测和定位故障是非常重要的。LOS (Loss of Signal)和BFD (Bidirectional Forwarding Detection)[9]被广泛用于检测链路故障。然而，使用LOS进行故障检测的时间可能只有几百毫秒，不能满足载波级网络的要求。BFD在一条路径的两个终端之间使用简单的hello-echo协议，时间消耗在[10]以下50ms。但是BFD无法定位路径中的故障链路，只能检测某条路径的故障，不能检测某条链路的故障。为了解决这一问题，[11]的作者提出了逐链路BFD机制，即在链路的两端而不是路径的两端应用BFD。通过实验，可以在10ms内完成故障检测。
  • 此外，在[12]和[13]中，提出了一种路径监控和故障定位机制。当一个监控周期覆盖了给定网络中的所有链路时，LSM (Link Status monitoring)包用于监控路径的活动状态。交换机收到LSM报文后，会将FLI报文发送给控制器。因此，当链路出现故障时，可以根据FLI数据包精确定位故障节点。[14]的作者提出了一种链路故障检测机制，利用多控制器之间的信息共享机制，用于检测交换机和控制器之间的控制通道故障。
  • 有了以上高效快速的检测方法，我们的故障恢复方法就可以实现，如何使故障检测后的反应时间最小化是我们论文的重点。
反应式故障恢复
  • 有两种机制可以处理链接故障:反应性和前瞻性。典型的反应性方法是[15]、[16]和[17]中提出的快速路径恢复方法。通过这种方法，交换机将拓扑变化事件通知给控制器。然后控制器计算并安装中断流的新路径。该进程耗时可能较大，可能导致丢包和业务中断。此外，[15]的作者还提出了一种改进的快速路径恢复方法，称为“预定恢复”。与快速恢复不同的是，备份路径是有优先级的预先规划，控制器可以为中断的流选择合适的备份路径。
  • 反应式恢复方法主要有两个缺点:一是由于远程控制器的参与，链路故障恢复的时间消耗不能满足载波级网络的需求，在[10]和[18]上进行了实验验证;其次，中断流的计算和安装路径可能给控制器带来沉重的负担，从而降低整个网络的性能。
主动故障恢复
  • 为了克服被动方法的缺点，主动方法使用静态预先计划的备份路径来重新路由中断的流，而不涉及控制器。而主动性方法的关键是建立一种切换机制，使旧路径的流入口失效，然后自动启用备份路径的流入口。在启用openflow的交换机中，除了“空闲超时”和“硬超时”等entry expiry timer外，没有这样的机制。因此，为了实现快速主动的链路故障，[1]提出了一种名为“自动拒绝”的方法，该方法可以检测到链路故障并禁用流入口。在[19]中，作者提出了一种流入口过期机制，用于在链路发生故障时自动删除旧路径的条目。在[2]中，作者使用Openstate来监控数据包和交换机端口状态，并在交换机端口故障时重新分配流到新的路径。然而，上述机制需要对交换机进行额外的修改，难以在实践中应用。Openflow v1.5通过实例化相同流条目的多个动作列表并根据链路状态[20]应用它们，提供了一种称为故障转移组表的快速故障转移机制。它不需要对Openflow协议进行任何修改，可以根据动作列表的活动状态实现自动切换。
  • 在上述切换机制的基础上，提出了几种主动切换方法。为了减少故障恢复时间，[1]的作者提出了一种改进的链路保护称为段保护。该方案不采用端到端备份路径，而是在每一跳上规划备份路径，使流可以在故障链路相邻的本地交换机上重路由，从而缩短故障恢复时间。但是，段保护需要大量的流入口来构建备份路径，需要复杂的机制来保持备份路径的活跃。
  • 为了减少流条目的数量在段保护,建立备份路径[3]的作者提出了一个新颖的设计被称为独立的瞬态平面(ITP),它包含链接,可以通过不同的备份路径路由共享流来使流条目的数量减少。此外，[7]的作者将[21]应用到基于openflow的网络中。通过附加的环路检测机制，可以在很低的开销下实现最大的保护覆盖。但这些方法仍然存在两个问题:一是流量入口的消耗量仍然较大，需要进一步优化;第二，备份路径规划过程中没有考虑链路状态，在重路由过程中可能会造成拥塞。
  • 为了解决拥塞问题，[13]提出了一种考虑链路利用率的备份路径优化方法。提出了优化模型并进行了评价。虽然使用优化后的备份路径可以减少拥塞的可能性，但由于静态路径不能根据网络状态自适应调整，因此效率仍然不够。
  • 在[2]中，作者提出了一种基于Openstate的多路径重路由(MPR)方法。该方法通过为每条链路规划多条备份路径，将流分配到备份路径，从而提高了网段保护能力，解决了拥塞问题。通过优化模型，该方法可以显著降低拥塞的可能性，但备份路径仍然是静态规划的，需要太多的流入口。
  • 在[4]中提出了一种基于源路由的SlickFlow方法。它将所有备份路径编码到包头中。当链路发生故障时，交换机查找备份路径并重新路由数据包。显然，这种方法给包头带来了太多的开销，而且还使用了静态备份路径。
  • 在[22]和[23]中，研究了IP和Openflow混合网络中感知拥塞的链路故障快速恢复方法。Openflow交换机已预先规划并安装了多条备份路径。当发生故障时，IP交换机将中断的报文直接转发给Openflow交换机。然后Openflow交换机选择正确的备份路径重新路由数据包。虽然这种方法可以很好地处理拥塞问题，但它在IP网络中比SDN网络更有效。
  • 综上所述，目前还没有一种方法可以同时满足第1节中提到的三个要求，其主要原因是SDN中流与路径流传输的紧密耦合。在我们的方法中，我们使用源路由来分离流和它的路径，使它能够在非常低的开销下实现感知拥塞的快速链路故障。

提出的链路快速故障恢复的方法

报文头编码
  • 近年来，源路由SDN (source routing SDN)由于可以在不丧失流量可编程性的同时减少SDN中使用的表项数量而受到了一些关注。然而，在最新的Openflow中还不支持源路由。因此本文对802.1q的VLAN tag进行修改，实现源路由。vlan_id和vlan_user_priority字段用于对路由信息进行编码。
  • 与传统的源路由不同，我们的下一跳信息分为两部分:输出端口号和备份路径标记。输出端口号编码在vlan_id(VID)字段中，备份路径标记编码在vlan_user_priority(VUP)字段中，如图1所示:
  • 802.1q的VLAN tag中VID字段的长度为12位，最多支持212个下一跳。VUP字段长度为3bit，最多支持8条备份路径。网络正常时，交换机通过匹配VID字段将报文沿工作路径发送;链路故障时，通过匹配VUP字段将报文引导到备份路径。
  • 此外，VLAN tag和源路由报头的长度可能会给报文带来开销，从而限制源路由SDN的可扩展性。但是，由于VLAN tag使用的比特数较少，可以进一步优化报文头的长度。我们相信凭借Openflow提供的任意匹配能力，在不久的将来可以开发出更短包头长度的源路由机制。

方法程序
  • 本文提出的基于拥塞感知的链路故障快速恢复方法包括备份路径规划、备份路径选择、链路故障快速恢复3个步骤。前两个步骤在控制器和边缘交换机中执行，最后一个步骤仅在链路故障时在核心交换机中执行。
  • 在备份路径规划过程中，控制器根据网络状态，找到能够以最小跳数提供足够可用带宽的最优备份路径集。此外，为了降低备份路径规划算法的复杂度和路径构建的流入口消耗，需要限制每条链路的备份路径数量。然后控制器标记每个备份路径，并将相应的流表安装到交换机。规划过程如图5所示:

案例
  • 考虑图8的拓扑结构，从S到T有两个工作流程，分别标记为flow1和flow2。流量1的传输速率为50Mbps，流量2的传输速率为30Mbps。工作路径为S→D→T。根据端口id识别交换机的每个端口，如图8所示，那么我们可以通过以下步骤演示我们的快速故障恢复方法:
  • 步骤1: 备份路径规划。有两个环节需要保护:S→DS→DS→D和D→TD→TD→T。我们假定每个链路使用两条备份路径。控制器根据网络情况，执行S→DS→DS→D、D→TD→TD→T两条链路的备份路径规划算法。假设经过备份路径规划过程，为S→DS→DS→D选择两条备份路径S→A→B→C→TS→A→B→C→TS→A→B→C→T和S→E→F→G→TS→E→F→G→TS→E→F→G→T，分别标记为psd,1p_{sd,1}psd,1​和psd,2p_{sd,2}psd,2​。同样地，D→TD→TD→T选择路径D→B→C→TD→B→C→TD→B→C→T和D→F→G→TD→F→G→TD→F→G→T，并被pdt,1p_{dt,1}pdt,1​和pdt,2p_{dt,2}pdt,2​标记。然后控制器在交换机S和D上安装流项，流项与报文的VUP字段匹配，更新报文头。
  • 步骤2: 选择备份路径。假设交换机S的备份路径psd,1p_{sd,1}psd,1​和psd,2p_{sd,2}psd,2​的可用带宽分别为70Mbps和60Mbps。在备份路径选择过程结束后，我们假设控制器根据备份路径的可用带宽和流的需求，在交换机S将flow1分配给路径psd,1p_{sd,1}psd,1​，将flow2分配给路径psd,2p_{sd,2}psd,2​。然后将flow1的第一跳信息组织为“1psd,11p_{sd,1}1psd,1​”。其中1为输出端口ID, psd,1p_{sd,1}psd,1​为备份路径ID。此外，如果备份路径标记为X，则流没有备份路径。在交换机D中经过相同的过程后，我们假设flow1的包头为“1psd,11pdt,11p_{sd,1}1p_{dt,1}1psd,1​1pdt,1​”，flow2的包头为“1psd,21pdt,21p_{sd,2}1p_{dt,2}1psd,2​1pdt,2​”。
  • 步骤3: 快速链接故障恢复。我们假设链路D→TD→TD→T在某个时间发生故障，然后快速恢复过程开始。我们以flow1为例。在交换机S，它检查最外层的VLAN标签为“1psd,11p_{sd,1}1psd,1​”。匹配后，交换机删除标签“1psd,11p_{sd,1}1psd,1​”，报文头由“1psd,11pdt,11p_{sd,1}1p_{dt,1}1psd,1​1pdt,1​”变为“1pdt,11p_{dt,1}1pdt,1​”。然后交换机S通过port1输出报文，因为在链路S→DS→DS→D没有链路故障。在交换机D，包的头是“1pdt,11p_{dt,1}1pdt,1​”，端口1失败。然后切换D查看备份路径ID为pdt,1p_{dt,1}pdt,1​。根据step1中安装的流表项，交换机D弹出VLAN标签“1pdt,11p_{dt,1}1pdt,1​”，并在交换机D处推送VLAN标签“0X”，“1X”，“3X”，表示备份路径1:D→B→C→TD→B→C→TD→B→C→T，重新检查报文，通过port3输出。同样的操作也会在flow2上执行，这样中断的流就可以绕过失败的链路。
